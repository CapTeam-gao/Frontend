# CapTeam Frontend

AI 기반 캡스톤 팀 자동 생성 및 관리 서비스 **CapTeam**의 프론트엔드 레포지토리다.

CapTeam은  
학생 설문 데이터와 AI 분석을 바탕으로 캡스톤 팀을 추천하고,  
관리자가 추천 결과를 검토·수정·승인한 뒤  
공지, 팀 채팅, 프로젝트 기획서, 캡스톤 일지까지 한 서비스 안에서 운영할 수 있도록 만든다.

---

## 프로젝트 개요

캡스톤 팀 구성 과정에서는 아래 요소를 함께 고려해야 한다.

- 희망 직군
- 기술 스택
- 구현 경험
- 협업 성향
- 개발 성향
- 선호 팀원
- 팀장 희망 여부

기존에는 이런 정보를 수기로 정리하고 팀을 배정해야 해서 시간이 오래 걸리고,  
팀 배정 이후에도 공지, 팀 관리, 일지 작성, 프로젝트 기획 같은 작업이 여러 곳에 흩어져 불편함이 있었다.

CapTeam은 이 흐름을

**학생 정보 수집 → AI 팀 추천 → 관리자 승인 → 프로젝트 운영**

구조로 하나의 서비스 안에 정리한 프로젝트다.

## 서비스 흐름

```txt
학생
로그인
 → 설문 미완료 시 설문 페이지 이동
 → 설문 작성
 → 학생 대시보드
 → 팀 생성 전에는 일부 프로젝트 기능 제한
 → 팀 생성 후 팀 채팅 / 프로젝트 기획서 / 캡스톤 일지 사용

관리자
로그인
 → 학생 설문 완료 여부 확인
 → 학년별 팀 생성
 → AI 추천안 검토
 → 필요 시 팀원 수동 교체
 → 팀 승인
 → 팀 관리 / 공지 / 일지 / 채팅 관리
```

## 주요 구현 포인트

### 1. 설문 기반 학생 정보 수집

- 희망 직군, 기술 스택, 구현 경험 수집
- 협업 성향 / 개발 성향 문항 구성
- 응답 점수 평균 계산
- 역문항 및 응답 일관성 기반 신뢰도 처리 구조 반영

### 2. AI 팀 추천 결과 검토 흐름

- 학년별 팀 생성 요청
- 추천안 결과 확인
- 역할 분포 / 배정 이유 확인
- 학생 수동 교체
- 최종 승인 후 실제 팀 확정

### 3. 팀 생성 이후 운영 기능 연결

- 팀 채팅
- 프로젝트 기획서 작성
- 캡스톤 일지 작성 / 조회
- 공지 관리

즉 CapTeam은 팀 생성에서 끝나는 서비스가 아니라,  
**팀 생성 이후 프로젝트 운영까지 이어지는 구조**를 가진다.

### 4. 공지 상세 분기 렌더링

- 일반 공지는 markdown 기반 상세 화면으로 렌더링
- 팀 배정 결과 공지는 `noticeType === "TEAM_RESULT"` 기준으로 전용 UI 렌더링
- 팀 배정 결과 공지에서는 팀장, 팀원 구성, 역할 요약을 한눈에 확인할 수 있다

### 5. 실시간 채팅

- STOMP + SockJS 기반 실시간 메시지 전송
- 학생 팀 채팅 / 관리자 채팅 관리 분리
- 팀장 권한 기반 채널 생성 / 수정 / 삭제
- 메시지 수정 / 삭제
- 파일 업로드 / 다운로드
- 이미지 미리보기
- 링크 자동 인식
- 읽지 않은 메시지 표시
- 온라인 / 오프라인 상태 표시

## 인증 / 권한 처리

- JWT 기반 로그인 처리
- `accessToken`을 Zustand + localStorage에 저장
- Axios interceptor로 요청 시 Authorization 헤더 자동 추가
- 토큰 만료 시 `refreshToken` 기반 재발급 처리
- `ProtectedRoute`로 학생 / 관리자 라우트 분리
- 팀 생성 상태, 설문 완료 여부에 따라 접근 가능한 페이지 분기

## 배포 방식

배포는 **Vercel** 기준으로 구성되어 있다.

### Vercel rewrite 설정

`vercel.json`

```json
{
  "rewrites": [
    {
      "source": "/api/:path*",
      "destination": "https://api.capteam.n-e.kr/api/:path*"
    },
    {
      "source": "/ws/:path*",
      "destination": "https://api.capteam.n-e.kr/ws/:path*"
    },
    {
      "source": "/(.*)",
      "destination": "/index.html"
    }
  ]
}
```

즉 production 환경에서는 프론트가 직접 외부 백엔드 URL을 들고 다니기보다  
`/api`, `/ws` 경로를 통해 프록시 방식으로 연결되도록 구성되어 있다.

### 환경 변수

개발 환경:

```env
VITE_BASE_URL=http://13.124.37.220:8888
```

배포 환경:

```env
VITE_BASE_URL=
```

production에서는 rewrite를 사용하므로,  
실제 배포 시 Vercel 환경 변수와 proxy 설정을 함께 확인해야 한다.
